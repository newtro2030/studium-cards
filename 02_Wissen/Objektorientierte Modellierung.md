Die objektorientierte Softwareentwicklung überwindet die Nachteile der [[Strukturierte Analyse und Entwurf|strukturierten Analyse]], indem Daten- und Verhaltensaspekte in **Objekten** zusammengefasst werden.

### Modellierung als Grundprinzip
- **Präskriptive** Modellierung: Modell gibt vor, wie System sein soll
- **Deskriptive** Modellierung: Modell beschreibt bestehendes System

Zweck von Modellen:
- Repräsentation von komplexen Systemen
- Diskussionsgrundlage mit [[Stakeholder|Stakeholdern]]
- Verifikation von gewünschten Eigenschaften
- Simulation von Systemverhalten

### Vision: Durchgängige Objektorientierung
In der **strukturierten Methode** werden unterschiedliche Werkzeuge pro Phase genutzt. In der **OO-Entwicklung** werden Objekte durchgängig über alle Phasen hinweg verwendet:

| Phase | Strukturiert (Daten) | Strukturiert (Verhalten) | Objektorientiert |
| :--- | :--- | :--- | :--- |
| Analyse | ER-Diagramme | Datenflussdiagramme | Klassendiagramme, Anwendungsfälle |
| Entwurf | Relationales DB-Schema | Funktionsbäume | Klassen-/Objektdiagramme, Sequenzdiagramme |
| Programmierung | SQL | Programm | OO-Programm (z.B. Java) |

### Realität: UML (Unified Modeling Language)
- Standardisierte Modellierungssprache für objektorientierte Analyse und Entwurf
- Datenaspekte: **Klassendiagramme und Objektdiagramme**
- Verhaltensaspekte: **Anwendungsfälle** → **Verhaltensdiagramme** (Sequenzdiagramme, ...)
- Integration beider Aspekte, aber nicht vollständig vereinheitlicht
